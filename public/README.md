This is Astro's static-asset passthrough directory (files here are copied byte-for-byte into `dist/` on build). It belongs only to the Astro project under `src/`.

It is unrelated to — and does not replace — the root-level `components/` directory, which is still used directly by the live, unmigrated root `*.html` pages. Do not copy site pages into this directory.

`js/`, `data/`, and `images/` here are **symlinks to the matching root directories** (e.g. `public/js -> ../js`), not copies — Astro pages need their relative-path references (`<script src="js/...">`, a runtime `fetch("data/...")`, `<img src="images/...">`) to actually resolve at build time, and a symlink keeps a single source of truth so editing a root file is automatically reflected for both the still-live legacy pages and the Astro build, with no risk of the two drifting out of sync. Add another symlink here the same way (not a copy) if a later migration phase needs another root directory inside the Astro build.

`functions/shared-firebase.js` is a **single-file symlink**, not a whole-directory one (`public/functions/shared-firebase.js -> ../../functions/shared-firebase.js`) — despite living under `functions/` (the Cloud Functions backend source), it's actually a client-side real-time Firestore listener that `results.html` loads directly. Don't symlink the whole `functions/` directory here — the rest of it is genuinely server-side code (uses `firebase-admin`, not meant to be publicly downloadable as static files).

`CNAME`, `robots.txt`, and `sitemap.xml` are also symlinks to the matching root files — once the site is deployed via the GitHub Actions Pages workflow (`.github/workflows/deploy-pages.yml`), `dist/` is the *entire* published artifact, so anything the live site needs (the custom domain mapping, crawler directives, the sitemap) has to be reachable from here, not just sitting at the repo root.
